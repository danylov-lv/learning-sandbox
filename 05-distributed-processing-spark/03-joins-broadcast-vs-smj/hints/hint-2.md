**`broadcast_enrich`.** `pyspark.sql.functions.broadcast(df)` wraps a DataFrame with a hint that forces the planner to broadcast it regardless of size estimates — use it on the reference-table side of each join, not the event side. Two sequential joins (events → sources, then that result → categories) each get their own hint; a plan with two broadcast joins in it will show `BroadcastHashJoin` more than once when you search the plan text. Read `sources.csv`/`categories.csv` with both `header` and `inferSchema` options so `region`/`vertical`/`source_id`/`category_id` come back as usable typed columns, not all strings.

**`force_sort_merge`.** Setting `spark.sql.autoBroadcastJoinThreshold` to `-1` doesn't make a side "not small" — it removes broadcasting as an option at all, for every join in that session's config scope, unless a join explicitly hints otherwise. That's the mechanism that guarantees sort-merge here even though a `groupBy("product_id", "source_id")` aggregate over one month is not enormous. For the correctness cross-check, build the join a second time with `F.broadcast(...)` on one of the two aggregated DataFrames — the hint wins over the `-1` threshold for that specific join, giving you a genuinely different physical plan computing the same logical join.

**`aqe_converts_join`.** The two plan captures in this function are on the *same* DataFrame object — you're not building two different queries, you're asking Spark to describe its plan for one query at two different points in that query's lifecycle: once before anything has run (the compile-time plan), once after an action has forced execution (the runtime-adapted plan). `get_plan(df, "formatted")` after the action reflects what the `AdaptiveSparkPlan` node actually decided, including the parts of the tree AQE rewrote — search that text for the literal marker AQE prints once it locks in a plan.
