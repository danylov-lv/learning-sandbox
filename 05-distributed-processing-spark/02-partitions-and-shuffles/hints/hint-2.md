Per-partition row counts are not something Spark hands you as a single number — you get at them by turning "which partition is this row physically in" into a column you can group on. `pyspark.sql.functions.spark_partition_id()` does exactly that: it's a column expression that evaluates to the physical partition index of the row it's attached to, at the point in the plan where you reference it. Attach it right after the repartition whose distribution you're measuring (not before — partition id upstream of a repartition describes the old layout, not the one you care about), then `groupBy(spark_partition_id()).count()` and collect the small result to the driver. Make sure every one of the `n_partitions` indices shows up in your final list, including ones that happen to hold zero rows — a partition that got no rows from a skewed key is itself part of the story, not a gap to drop.

`spark.sql.shuffle.partitions` is scoped narrowly: it sets the partition count of the *output* of an operation that shuffles — `groupBy(...).agg(...)`, `distinct()`, a shuffle join, `repartition()` without an explicit column list argument mismatch, etc. It says nothing about a DataFrame that was never shuffled (a fresh read keeps whatever partition count the file layout gave it) and nothing about `coalesce()` (which by design avoids the shuffle machinery this config governs). If you set it and then look at the partition count of something that never went through a shuffle, you're not testing what you think you're testing.

For salting: the problem with `groupBy("source_id")` under skew is that every row for the dominant source hashes to the same bucket, no matter how many buckets you have — more partitions doesn't help because the hash of a fixed key value is fixed. The fix is to make the key itself less uniform than the data by attaching randomness to it before the shuffle: derive a new column that's an integer drawn uniformly from `[0, n_salts)` per row (a `rand()`-based expression, cast to int, is one way; a hash of something row-unique modulo `n_salts` is another), then build a composite key out of `(source_id, salt)` — concatenating the two into one column is enough, they don't need to stay separate fields. Partitioning on that composite key spreads one dominant source's rows across up to `n_salts` different physical partitions instead of one. That only gets you halfway, though: your aggregate is now computed per `(source_id, salt)` pair, which is not the number you actually want. Recovering the true per-source result needs a second aggregation pass, over the first pass's output, grouped by `source_id` alone, summing across the salt buckets you just split it into.
