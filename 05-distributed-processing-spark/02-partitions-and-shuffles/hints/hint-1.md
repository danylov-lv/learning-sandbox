A DataFrame's partition count is a physical property, independent of how many rows are in it or what the data looks like logically. It's set when you read (roughly: one partition per input file, or per file-split if a file is large — Spark decides based on file size and `spark.sql.files.maxPartitionBytes`), and it changes at specific points afterward: any shuffle re-partitions into `spark.sql.shuffle.partitions` partitions (200 by default — yes, even for a tiny DataFrame, unless you've configured it differently), and `repartition()`/`coalesce()` change it explicitly.

`repartition(N)` always triggers a full shuffle, even if you're increasing the count or going from an already-even distribution to another even one — it doesn't know your data is already fine, it just redistributes by hash (or by a column, or randomly, depending on the arguments you give it). `coalesce(N)` is different: when `N` is smaller than the current partition count, it can merge existing partitions locally, on the same executor, without moving any data across the network — no shuffle, no `Exchange` in the plan. That's exactly why `coalesce` can't increase partition count (there's nothing to locally split), while `repartition` can do either.

Skew is not about partition *count* — you can have exactly the right number of partitions and still have one partition holding 30% of the rows, if the key you partitioned on happens to have one dominant value. Fixing skew means changing what you partition *on*, not how many partitions you ask for.
