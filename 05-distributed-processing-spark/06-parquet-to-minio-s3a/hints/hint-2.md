**File-count control.** The operation is `repartition`. Specifically, `repartition("month")` (or `repartition(<some N>, "month")` if you also want to cap parallelism) before `.write.partitionBy("month")`. Repartitioning by the same column you're about to partition the write by shuffles the DataFrame so that every row for a given month ends up in exactly one in-memory partition, and only one. When the writer then fans out by `month`, each month's single partition produces exactly one file in that month's directory. Skipping this step is the single most common cause of the small-files problem in object-store lakes — it's not an obscure gotcha, it's the default behavior of every naive `partitionBy` write.

**Reading the plan.** Build the filtered DataFrame (`spark.read.parquet(dest).filter(col("month") == month)`) and capture its plan with `harness.common.get_plan(df, "formatted")` *before* calling any action on it — same discipline as task 01's `pushdown_comparison`. Look at the `Scan parquet` node specifically. There is a field there that lists which predicates were used to decide *which files/directories to even list*, as opposed to a field that lists predicates applied to rows *after* they're read. Those are two different mechanisms in Spark's plan output, and only one of them applies at the partition-directory level for a `partitionBy`-written table. The field name you're looking for describes exactly that: filtering on the partitioning column(s) to skip whole directories before any row is read.

**Proving it at runtme, not just in the plan.** A plan telling you pruning *should* happen isn't the same as observing that it *did*. `pyspark.sql.functions.input_file_name()` gives you, per row, the actual file path Spark read that row from. Select it (aliased), `.distinct()`, `.collect()` on the filtered DataFrame, and count how many distinct paths came back. If your write step did its job (one file per month) and pruning did its job, this number can only be 1.
