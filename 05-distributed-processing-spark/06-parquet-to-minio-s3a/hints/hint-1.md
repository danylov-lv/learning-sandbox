Think about what `write.partitionBy("month")` actually controls versus what it doesn't. It tells Spark's writer which column to fan output out by on disk — one subdirectory per distinct value. It says nothing about how many files land inside each of those subdirectories. That's decided by something else entirely: how many in-memory partitions of the DataFrame being written contain rows for a given month, at the moment the write executes. If the DataFrame still has its "however Spark happened to split the input" partition count and layout when you call `.write`, each of those partitions writes its own file into whichever month directories its rows belong to. A month whose rows are scattered across forty partitions gets forty files.

The fix is not a writer option — it's something you do to the DataFrame *before* you call `.write`. There's an operation you've already used in earlier tasks in this module for controlling exactly this kind of thing: how many partitions a DataFrame has, and which rows land in which partition, based on a key.

For the pruning task, don't reason from memory about what a Spark "partitioned Parquet scan on s3a" plan looks like — the module's other tasks have already told you not to guess at plan text, and that advice applies doubly here since this is the first time you're reading a plan for an object-store source rather than a local one. Capture the plan, print it while developing, and read the `Scan parquet` node's fields the same way you did for pushdown filters in task 01.
