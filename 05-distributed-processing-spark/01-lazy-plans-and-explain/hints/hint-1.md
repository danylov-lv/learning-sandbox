Think back to Postgres `EXPLAIN`: you never had to worry about "did this query actually run yet" — issuing a query ran it, full stop (unless you asked for just the plan). Spark's DataFrame API splits that in two on purpose. A DataFrame is a description of work, not a result. `.filter()`, `.select()`, `.groupBy()`, `.withColumn()` — none of these touch data. They build up a logical plan, tree by tree, and return instantly regardless of how large the underlying data is, because "how large" isn't even known yet in any exact sense.

An action is the thing that actually asks Spark to produce a concrete answer: `.count()`, `.collect()`, `.show()`, `.write.parquet(...)`. Only at that point does Spark compile the accumulated plan into stages and tasks and launch executors to run them. This is why "my code is slow" debugging in Spark almost always means "my code launched an action that has to do more work than I expected" — the transformations leading up to it were free to write, but not free to run.

Before writing any code, go build the mental model: for each function you're implementing, ask "does this line touch data, or does it describe work?" You'll use that distinction directly for job counting, and it explains why the wide-vs-narrow plan and the JSONL-vs-Parquet plan questions have real, inspectable answers — the plan tree records exactly what work was described, before any of it runs.
