A window function doesn't reduce the number of rows the way a `groupBy` does — every input row keeps its place, and each row additionally gets to see some aggregate or positional fact about the group of rows it belongs to (its rank within the group, the value of the row before it in some ordering, a running total, whatever). That "sees the rest of its group" part is exactly what makes it expensive: Spark can't compute a row's rank until it has gathered every other row in the same partition and sorted them, and it can't do that without a shuffle by the partition key first.

Before writing any code, work out for each of the four functions: does it need to see every row in a group ranked against each other (a genuine window question), or does it only need one summary number per group (a `groupBy` question wearing a window-shaped costume)? `top_n_per_source` and `price_change_per_product` are the former. Part of `window_vs_aggregate_plans` is deliberately the latter, computed both ways so you can see the cost difference in the plan.

Separately: go look at a few entries in `data/ground-truth.json`'s `top_n_per_source.by_source`. At least one source's list contains the same `product_id` twice, at two different prices. Before you pick a ranking function, make sure you understand why that's not a bug in the ground truth and what it implies about whether "top n" here means "top n distinct products" or something else.
