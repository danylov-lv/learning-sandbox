**`top_n_per_source`.** `Window.partitionBy("source_id").orderBy(F.desc("price"), F.desc("product_id"))`. `ground-truth.json` was built by sorting each source's rows by that exact key and slicing the first n — it never groups by product_id first, and it never collapses ties into one shared position. That second fact is the one that decides between `rank()`/`dense_rank()` and `row_number()`: the former two let every row tied for a boundary rank "through" a `<= n` filter, so a source with a 3-way tie at the boundary would come back with more than n rows. `row_number()` never does that — it always returns exactly n rows (given at least n to begin with), which is what a Python-side `sorted(...)[:n]` slice (which is what generated the ground truth) also always does.

**`price_change_per_product`.** `Window.partitionBy("product_id").orderBy("captured_at")`, then `F.lag("price").over(w)` and `F.lag("captured_at").over(w)` computed on *all* http_status==200 rows for that product — do this before you filter down to just the latest row per product, since the lag needs the row that comes immediately before the latest one, which only exists in the unfiltered per-product sequence. To find "the latest row per product" afterward, either a second window (`row_number()` over the same partition ordered descending, filtered to 1) or a `groupBy(product_id).agg(max(captured_at))` + join both work — the validator only checks the final output, not which of those two you used internally.

**`window_vs_aggregate_plans`.** The windowed branch is `top_n_per_source`'s n=1 case, minus the source-list-comparison plumbing — same partitionBy/orderBy, filtered to `rn == 1`. The aggregate branch is a one-liner: `events_df.filter(http_status==200).groupBy("source_id").agg(F.max("price"))`. Capture each with `get_plan(df)`. You do not need `.limit()` or `.collect()` before capturing a plan — `explain()`/`get_plan()` works on an unmaterialized DataFrame just fine, though calling `.collect()` first (to build the return dict's price-by-source maps) doesn't hurt either capture.
