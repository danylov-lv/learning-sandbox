For the malformed-line problem: nothing you pass to `pl.scan_ndjson` or `pl.read_ndjson` as a *reader option* will get you past a genuine JSON syntax error — that door is closed, verified against 1.38.1. What's open is filtering the bad lines out before they ever reach polars' JSON parser. You have the raw file(s) as plain text; a valid JSON object line in this dataset always starts with `{` and ends with `}` (after stripping the newline). You don't need a full JSON parse to decide whether a line is a parse candidate — a much cheaper structural check on the raw bytes gets you there, and you can hand the survivors to polars however you like (write them to a cleaned file, or feed them to `scan_ndjson`/`read_ndjson` via an in-memory buffer — both accept file-like objects, not just paths).

For `load_events`'s dedup step: polars' `.unique()` (or `.unique(keep="any")`) works over a full row, struct columns included — you don't need to flatten `attrs` first.

For `monthly_rollup`: `captured_at` is a plain string column; slicing its first 7 characters gets you a `"YYYY-MM"` key without parsing a date type at all. `group_by(...).agg(...)` with a conditional inside the aggregation expression (something that evaluates to null/excluded when `http_status != 200`) gets you both the all-rows count and the status-200-only price sum in a single pass over the grouped data.

For `top3_per_source`: this is the same shape as a Spark window-function top-N — filter first, sort by the full tie-break key (price, then product_id, both descending), then take the first 3 rows per source group. Polars has more than one way to express "first N rows per group after a sort"; look at what `group_by(...).head(n)` needs to see beforehand for its output to already be in the right order, versus building a rank column explicitly.
