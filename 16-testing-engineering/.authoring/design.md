# Module 16 design notes — OFF-LIMITS TO THE LEARNER

This directory (`.authoring/`) is committed but must never be read by the
learner before attempting a task — it documents planted mutant themes,
which is exactly the answer key. Read it after finishing a task, if at all,
same rule as every other module (see repo `CONVENTIONS.md`).

This file is the design contract for task-authoring agents building tasks
01–07 of `16-testing-engineering`. It documents infrastructure that is
already built (`harness/`, `pyproject.toml`, `uv.lock`) and the decisions
behind it, so a task agent does not have to re-derive the harness contract
or rediscover the Windows gotchas below.

## The inversion this module is built on

Every other module in this repo has the learner write an implementation
that is tested. Here it is reversed: the sandbox provides a GIVEN, CORRECT
`src/impl.py`; the learner writes the TEST SUITE. Grading is mutant-killing,
not output-checking. Two independent gates, both enforced by
`harness.mutation.grade()`:

1. **Green-on-correct** — the suite must fully pass against the correct
   implementation, and must collect at least `min_tests` tests. This
   rejects `assert False` (fails "must pass") and rejects nothing collected
   / collection errors (pytest exit code != 0, same gate).
2. **Kills every mutant** — the suite must FAIL against each file in the
   task's mutant bank. A mutant the suite still passes against "survived"
   and the task is rejected, naming the surviving mutant ids.

`impl.py` is NOT a reference solution to the task (the task is "write
tests", not "write impl") so committing it in the open is fine and
necessary — the learner is meant to read it.

## Canonical task directory layout

```
NN-task-name/
  README.md
  src/
    impl.py            # GIVEN correct implementation — learner reads this, does not edit it
    sut.py              # generated by harness.mutation.write_sut_shim() — learner tests import THIS
  tests/
    test_*.py           # learner-authored — this is the actual deliverable
    validate.py          # thin call into harness.mutation.grade(...)
  hints/
    hint-1.md / hint-2.md / hint-3.md
  NOTES.md

.authoring/
  design.md              # this file
  mutants/
    NN-task-name/
      m01.py              # generic filename — the docstring at the top carries the spoiler, learner never opens this
      m02.py
      ...
```

- No `src/__init__.py` and no `tests/__init__.py` are needed. `src` is
  imported as an implicit Python 3 namespace package.
- No `conftest.py` is needed for import-path purposes. Tests are always
  invoked as `python -m pytest <paths> -q` (see `harness/mutation.py`'s
  `run_suite`), and `python -m X` inserts the current working directory as
  `sys.path[0]` **before** pytest even starts. `cwd` is always the task
  root when `grade()` runs the suite, so `from src.sut import name` just
  resolves. A task may still add its own `conftest.py` for pytest fixtures
  (e.g. a session-scoped Postgres testcontainer fixture in tasks 03/07) —
  that is orthogonal and fine, just not required for path resolution.
- Mutant banks live under `.authoring/mutants/NN-task-name/`, never inside
  the task directory itself — the learner's `python -m pytest` invocation
  never has that directory on any search path, so there is no risk of an
  IDE or curious learner stumbling onto it via imports. `grade()` is
  handed the mutant dir explicitly by `validate.py`.
- Mutant filenames must be generic (`m01.py`, `m02.py`, ... — never
  `m03_off_by_one.py`). The human-readable bug description goes in a
  **top-of-file module docstring only** (see the two example mutants
  further down) — that keeps `not_passed()`'s survivor list
  (`"2 mutant(s) survived: m03, m07"`) spoiler-free even though filenames
  are printed straight to the learner.

## The SUT shim mechanism

Learner tests never import `src.impl` directly. They import `src.sut`,
which is a **generated** file (`harness.mutation.write_sut_shim`) that:

1. Reads `SUT_IMPL_PATH` from the environment.
2. If set, loads that file as a standalone module via
   `importlib.util.spec_from_file_location` + `module_from_spec` +
   `exec_module` (this is how a mutant file — living outside the `src`
   package, anywhere on disk — gets loaded without needing to be a real
   package member).
3. If unset, falls back to `from . import impl` (the correct
   implementation, same package).
4. Re-exports every public name from the loaded module into its own
   globals: `globals().update({k: v for k, v in vars(_mod).items() if not
   k.startswith('_')})`.

Verified in the toy proof (below): `from src.sut import clamp` binds
correctly under all three conditions (`SUT_IMPL_PATH` unset → correct impl;
set to each of two mutant files) in three separate fresh subprocesses, with
no stale-import contamination between them. Each `run_suite()` call is a
brand-new `python -m pytest` subprocess specifically so Python's module
cache never lets one mutant's import linger into the next.

Call `harness.mutation.write_sut_shim(task_root / "src" / "sut.py")` once
while authoring a task, and commit the generated file — it does not change
at grading time, only the env var does. Pass `public_names=[...]` instead
of the default (export-everything-public) behavior if `impl.py` has helper
names that should not leak into the learner-facing surface.

## `harness/mutation.py` — public API

```python
def run_suite(
    test_paths: list[str], impl_path: str | None, cwd: str,
    extra_env: dict | None = None, timeout: int = 300, fail_fast: bool = False,
) -> SuiteResult
```
Runs `[sys.executable, "-m", "pytest", *test_paths, "-q"]` (`-x` appended
iff `fail_fast=True`) as a fresh subprocess, `cwd=cwd`, with
`SUT_IMPL_PATH` set to `impl_path` (or unset if `impl_path is None`).
Returns `SuiteResult(passed, returncode, collected, output_tail,
timed_out=False)`.

- **Always `sys.executable`, never a bare `"python"` string** — see the
  Windows PATH gotcha below, this is load-bearing.
- `collected` is parsed from the **final summary line** (e.g. `"3 passed,
  1 failed in 0.42s"`), not from a `"collected N items"` header — `pytest
  -q` does not print that header at all. Only the last 5 lines of combined
  stdout+stderr are scanned, to avoid false hits inside a learner's
  traceback text.
- A `subprocess.TimeoutExpired` becomes `SuiteResult(timed_out=True,
  passed=False, returncode=-1, ...)` rather than propagating.

```python
def grade(
    test_paths: list[str], correct_impl: str, mutant_dir: str, cwd: str,
    min_tests: int = 1, required_kill_ratio: float = 1.0,
    extra_env: dict | None = None, timeout: int = 300,
) -> None
```
The full protocol; this is meant to be the entire body of a task's
`tests/validate.py`. Ends the process itself via `harness.common.passed` /
`not_passed` — it never returns to a caller that expects a value.

1. Runs the suite once against `correct_impl` (`fail_fast=False`, so a full
   `collected` count is available). Not passing, or `collected < min_tests`,
   is `NOT PASSED` with a specific reason for each case.
2. Runs the suite once per mutant file in `sorted(Path(mutant_dir).glob("*.py"))`
   (skipping `__init__.py` and any `_`-prefixed file), `fail_fast=True` for
   speed. A mutant the suite still passes against is a survivor.
3. A per-mutant timeout counts as **killed** (the mutant broke the suite
   badly enough to hang it — that is still "the suite would have caught
   this"), but is called out by id in the success detail line
   (`"killed 7/7 mutants (1 counted as killed via timeout: m04)"`) so a
   task author notices and can consider redesigning a mutant that hangs
   instead of failing fast.
4. `killed / total < required_kill_ratio` → `NOT PASSED: "N mutant(s)
   survived: <ids>. Your suite does not catch these regressions."` Full
   kill → `PASSED` / `killed N/N mutants`.

A minimal `validate.py` for a task looks like:

```python
import sys
from pathlib import Path

TASK_ROOT = Path(__file__).resolve().parents[1]
MODULE_ROOT = TASK_ROOT.parent
sys.path.insert(0, str(MODULE_ROOT))
sys.path.insert(0, str(TASK_ROOT))

from harness.mutation import grade  # noqa: E402

if __name__ == "__main__":
    grade(
        test_paths=["tests/test_parser.py"],
        correct_impl=str(TASK_ROOT / "src" / "impl.py"),
        mutant_dir=str(MODULE_ROOT / ".authoring" / "mutants" / TASK_ROOT.name),
        cwd=str(TASK_ROOT),
        min_tests=4,
    )
```

(`sys.path` insertion of `MODULE_ROOT` + `TASK_ROOT` mirrors the existing
convention already used by every other module's `tests/validate.py` in
this repo, e.g. `13-scraping-at-scale/01-hostile-target-recon/tests/validate.py`.)

## `harness/common.py` — public API

- `not_passed(reason: str) -> NoReturn` — prints `NOT PASSED: {reason}`,
  `sys.exit(1)`.
- `passed(msg: str = "") -> None` — prints `PASSED`, then `msg` on its own
  line if given.
- `guarded(fn)` — decorator for a validator `main()`; any uncaught
  exception becomes one `NOT PASSED: <last traceback line>` instead of a
  raw traceback. `_last_line(exc)` extracts that line via
  `traceback.format_exception_only`.

`harness.mutation.grade()` already ends the process itself (it's the
terminal call in `validate.py`), so `guarded`/`main()` wrapping is only
needed for tasks whose `validate.py` does extra checks *around* `grade()`
(e.g. task 07's CP3, which also checks `DESIGN.md` content — see task list
below) or for non-mutation-graded tasks in other modules.

## Mutation tool for task 06: `mutmut` does NOT run on native Windows

`mutmut>=3` was tried first (per the module's original dependency plan).
Confirmed live on this machine: `uv run mutmut --help` prints exactly

```
To run mutmut on Windows, please use the WSL. Native windows support is
tracked in issue https://github.com/boxed/mutmut/issues/397
```

and refuses to do anything else. It was removed from `pyproject.toml`.

**`cosmic-ray>=8.4.6` was tried next and works**, with one Windows-specific
fix required. It is now the pinned dependency for task 06 (and available to
any other task that wants a real mutation-testing tool rather than the
custom `harness.mutation` engine).

### The Windows gotcha: never use a bare `"python"` in `test-command`

`cosmic-ray`'s config (`[cosmic-ray] test-command = "..."`) is a shell
command string it subprocesses per mutant. Using `"python -m pytest ..."`
looked reasonable but **silently ran the wrong interpreter** on this
machine: `subprocess.run(["python", ...])`'s executable search resolved to
a *different* `python.exe` than the project's own `.venv` (a base
`cpython-3.14` under `AppData\Roaming\uv\python\...`), which does not have
`pytest` installed. The result: cosmic-ray reported every single mutant —
and even the unmutated baseline — as "killed", because the test command
itself failed to run at all (`ModuleNotFoundError: No module named
pytest`), not because any test failed. This is invisible unless you check
`cosmic-ray baseline` (which does fail loudly) or inspect `cosmic-ray dump`
output closely: `"output": ""` on every job was the tell.

**Fix**: build `test-command` with the project's own interpreter path,
resolved via `sys.executable` at config-generation time (forward slashes,
since it lands inside a TOML string):

```python
py = sys.executable.replace("\\", "/")
test_command = f'"{py}" -m pytest test_module.py -q'
```

Verified end to end on this machine (`cosmic-ray init` → `cosmic-ray exec`
→ `cosmic-ray dump`) with a two-function, 8-line toy module and two learner
suites:

- **Weak suite** (only tested one of the two functions): 11/27 mutants
  killed, 16 survived — all 16 survivors were mutations inside the
  untested function, exactly as expected.
- **Strong suite** (tested both functions, all boundaries): 25/27 killed,
  **2 survived** — both were `core/ReplaceComparisonOperator_*_Is`
  operators (`x < lo` mutated to `x is lo`). This is a genuine equivalent
  mutant for small-int test values: CPython caches small ints (-5..256),
  so `x is lo` and `x == lo` agree for every value the test suite used,
  even though `is` is semantically unrelated to `<`. This is a real,
  reproducible teaching moment worth keeping in mind when task 06 sets its
  required kill ratio / picks its target module — either exclude
  identity-comparison operators for that task, or pick example values
  outside the small-int cache, or accept that task's kill ratio need not
  be literally 1.0 for the built-in `core/*_Is`/`*_IsNot` operators
  specifically.

### How task 06 should invoke it (exception to the `grade()` protocol)

Task 06's validator does **not** call `harness.mutation.grade()` — the
spec explicitly grades it by the real tool's survivor count instead. The
recommended flow for that task's `validate.py`:

1. Generate a `cosmic-ray` TOML config pointing `module-path` at the
   task's target module and `test-command` at
   `f'"{sys.executable}" -m pytest {test_file} -q'` (see gotcha above).
2. `subprocess.run([sys.executable, "-m", "cosmic_ray.cli", "init", cfg, session_db])`
   then the same for `"exec"`. (Or shell out to the `cosmic-ray` console
   script directly, resolved via `Path(sys.executable).parent /
   "cosmic-ray[.exe]"` — either works; the module-form is more robust to
   PATH weirdness, consistent with the `run_suite` gotcha above.)
3. Parse `cosmic-ray dump session_db` (JSON-lines output, `[job, result]`
   pairs) — a result has `"test_outcome": "killed" | "survived"`. Assert
   zero `"survived"`, else `not_passed()` listing the survived
   `operator_name`s (safe to show — task 06's whole point is running the
   tool and reading survivor output, unlike the spoiler-sensitive custom
   mutant banks of the other tasks).
4. Also re-run `cosmic-ray baseline` first and treat a failing baseline as
   an internal-error `not_passed()` (it should never fail if the given
   module/tests are wired correctly — a failing baseline on task 06 means
   the test-command itself is broken, e.g. the Windows gotcha above, not
   that the learner did something wrong).

## Task list and mutant themes

One-line mutant THEME per task — what kind of bug each task's mutant bank
should encode, chosen so a trivial/lazy test suite (e.g. "call the function
once, check it doesn't crash") cannot kill them. This is the spoiler
contract task-authoring agents work from; mutant bank files still need
generic names (`m01.py`, ...) with the theme only in each mutant's
docstring.

| # | Task | Mutant themes |
|---|------|----------------|
| 01 | property-based-parsing | Dropped currency symbol; thousands/decimal-separator confusion (`1.234,56` vs `1,234.56`); sign/abs bug (negative price silently made positive or vice versa); silent failure returning `None` instead of raising on malformed input. |
| 02 | stateful-and-metamorphic | Eviction order wrong (FIFO instead of LRU, or vice versa); TTL boundary off-by-one (expires one tick early/late); capacity off-by-one (holds N+1 or N-1 items); `get()` fails to bump recency (or bumps it when it shouldn't, e.g. on a miss). |
| 03 | integration-postgres-testcontainers | Wrong `ON CONFLICT` target column (upsert not actually idempotent); missing `commit()`/transaction wrapping (silent data loss on error); pagination off-by-one (boundary row skipped or duplicated across pages); watermark comparison `>` vs `>=` (reprocesses or skips the boundary row). |
| 04 | integration-redis-testcontainers | TTL never set (key leaks forever); non-atomic check-then-set (race window between the check and the write); wrong key namespace (cross-tenant/cross-key collision); boundary count off-by-one in a rate limiter or dedup filter. |
| 05 | contract-tests-api | Renamed or dropped response field; wrong HTTP status code on an error path; `next_cursor` present when it should be omitted at the last page (or vice versa); type drift (id returned as `str` instead of `int`, or vice versa). |
| 06 | mutation-testing-taste | N/A — reflexive task, no custom mutant bank. Given module ships with a GIVEN weak-but-green suite; learner runs the real tool (`cosmic-ray`, see above) and adds tests until the survivor count is zero. Graded by the tool's own survivor count, not `harness.mutation.grade()`. |
| 07 | capstone-scrape-to-serve-test-suite | Large combined bank spanning every layer above: pure-parser bugs (CP1, same themes as 01/02) + DB/cache/API bugs (CP2, same themes as 03/04/05). CP3 adds no new mutants — it re-runs CP1+CP2 as subprocesses after `DESIGN.md` is filled in, and both must still be green. |

## Toy proof (deleted, not committed)

Built at `.authoring/_toy/` (a `clamp(x, lo, hi)` correct impl, two
mutants, a generated `src/sut.py`, and a `tests/validate.py` calling
`grade()`), exercised with three throwaway learner suites, then the whole
`_toy/` directory was deleted. Observed outcomes:

- **Good suite** (5 tests covering both boundaries and both mutants'
  actual behavior change): `PASSED` / `killed 2/2 mutants`, exit 0.
- **Empty suite** (`tests/test_learner.py` empty): `NOT PASSED: your tests
  must pass against the correct implementation (pytest exit code 5): ...
  no tests ran in 0.19s`, exit 1.
- **`assert False` suite**: `NOT PASSED: your tests must pass against the
  correct implementation (pytest exit code 1): ...`, exit 1.

One real bug caught during this proof, worth remembering when writing
mutants for tasks 01–05/07: a naive "off-by-one" mutant (`x < lo` →
`x <= lo` in a clamp function) can be a silent **equivalent mutant** — at
`x == lo` both branches return `lo` either way, so no test can distinguish
them by output. The toy's `m01` was rewritten to change the *returned
value* (`return lo + 1`) instead of the comparison operator, which is
unambiguously killable. Task-authoring agents should mentally execute each
mutant against a couple of concrete inputs before committing it to a bank,
not just eyeball the diff.
