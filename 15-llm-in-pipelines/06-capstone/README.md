# 06 — Capstone: End-to-End Enrichment Pipeline

## Backstory

Every earlier task in this module built one pipeline component in
isolation: pull structured fields out of hostile HTML (t02), classify and
enrich a diluted-signal record (t03), cluster near-duplicate title variants
by embedding similarity (t04), and answer a question by retrieving from a
small doc corpus (t05). A real catalog-enrichment job doesn't get to run
those one at a time on hand-picked inputs and call it done — it runs all of
them, back to back, unattended, on a batch of records that is never
perfectly clean, against a model that occasionally returns garbage instead
of the JSON you asked for. Something has to decide, per record, "is this
good enough to ship," and route the rest somewhere a human (or a retry job)
can look at it later instead of poisoning the clean output.

This capstone builds that: one pipeline that composes extraction,
classification+enrichment, and dedup, gated by a confidence check that
routes unreliable records to quarantine instead of the clean catalog, plus
a small RAG feature that answers a question about a specific product by
retrieving from the catalog itself.

## What's given

- `src/pipeline.py` — stubs for `extract_record`, `classify_record`,
  `dedup_cluster`, and `run_pipeline`, each `raise NotImplementedError`
  with a docstring spelling out the exact contract (return keys, ordering
  guarantees, and the "must never raise" requirement every one of them has
  to satisfy).
- `src/explain.py` — stubs for `render_catalog_doc` and `explain_product`,
  the RAG "explain this product" step.
- `DESIGN.md` — an unfilled template with six sections for CP3.
- Three checkpoint validators: `tests/validate_cp1.py`, `validate_cp2.py`,
  `validate_cp3.py`.
- The same shared inputs every other task in this module uses:
  `data/extraction.json`, `data/classification.json`, `data/dedup.json`
  (gold-stripped, generated by `uv run python generate.py` from the module
  root), and `generate.build_catalog(seed, 1.0)` for the product catalog
  `explain_product` searches over.

## What's required

### CP1 — steady state (`validate_cp1.py`)

**Build:** in `src/pipeline.py`, implement `extract_record`,
`classify_record`, `dedup_cluster`, and `run_pipeline`. In `src/explain.py`,
implement `render_catalog_doc` and `explain_product`. Read every docstring
closely — the exact return keys and per-item ordering guarantees are the
contract the validator checks against.

**Checked:** `run_pipeline` run over the CLEAN extraction/classification/
dedup sets — extraction per-field accuracy, classification macro-F1 (and
brand accuracy), and dedup pairwise F1 must each clear their own threshold.
The quarantine rate on this clean input must stay LOW (the gate shouldn't
be rejecting good records just because it can). `explain_product` is
checked over five deterministic product questions for citation hit-rate
and answer-contains-fact rate.

### CP2 — chaos (`validate_cp2.py`)

**Build:** nothing new — the SAME `src/pipeline.py` from CP1. This
checkpoint exists to prove `extract_record` / `classify_record` /
`run_pipeline` actually honor their "must never raise, must self-assess
reliability" contract instead of only working on clean input.

**Checked:** the validator itself corrupts (truncates) a chunk of the
extraction HTML and wraps your client so a fraction of `generate`/`chat`
calls return non-JSON junk. `run_pipeline` must not crash, must still
return the full expected shape, and its catalog/quarantine routing must
show REAL signal: most records that land in the clean catalog must
actually be correct ("catalog precision"), and most records that are
actually wrong must land in quarantine ("quarantine recall"). Neither an
empty quarantine (gate never engages) nor an empty catalog (gate rejects
everything) passes this checkpoint.

### CP3 — design memo + regression gate (`validate_cp3.py`)

**Build:** fill in every section of `DESIGN.md` — pipeline architecture,
the quality/confidence gate, chaos handling and graceful degradation, the
RAG explain-product step, metrics and thresholds, and scaling to
production.

**Checked:** `DESIGN.md` is gated first (every required section present,
filled with real content past the shipped placeholder, a minimum length
per section, and grounding keywords proving the memo is about THIS
capstone). Only once that passes does the validator re-run CP1 and CP2 as
subprocesses and require both to still print `PASSED`.

```bash

uv run python 06-capstone/tests/validate_cp3.py
```

The task is complete when all three print `PASSED` and exit 0. Every
LLM-dependent validator calls `require_client()` first, so an unreachable
Ollama server is reported as an actionable infra message, not a confusing
metric failure. CP1/CP2 each make on the order of 130+ live model calls
and 55+ embedding calls, so a single run can take several minutes — that's
expected, not a sign something's stuck.

## Topics to read up on

- Composing several LLM-backed stages into one pipeline without letting one
  stage's failure abort the whole batch
- Confidence scoring and quality gates: deciding, per record, whether
  automated output is trustworthy enough to ship unreviewed
- Graceful degradation under partial failure (malformed input, a
  degraded/unreliable model backend) vs. failing the whole job
- Precision/recall trade-offs in a routing decision (quarantine too much
  and you've built an expensive no-op; quarantine too little and the gate
  isn't doing anything)
- Retrieval-augmented generation grounded in structured records rather than
  prose documents — rendering a record to a searchable text form
- Why a validator should grade a routing DECISION's outcome (did the right
  records end up in the right bucket) rather than trusting a component's
  own self-reported confidence number as ground truth
