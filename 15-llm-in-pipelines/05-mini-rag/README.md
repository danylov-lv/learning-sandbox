# 05 -- Mini RAG

## Backstory

Someone on the team keeps asking the same handful of questions in chat --
"what port does Ollama listen on again?", "how many hint files does a task
get?", "what's the exit code for a passing validator?" -- questions that are
all answered, in writing, somewhere in the team's internal docs. Rather than
keep answering them by hand, you're building a small lookup tool: point it
at the docs, ask it a question in plain English, and it finds the right
passage and answers from it, telling you which doc it got the answer from
so you can go check the source yourself if you don't trust it.

That's retrieval-augmented generation in miniature. The model doesn't need
to have memorized the docs -- it needs a way to find the right few sentences
out of everything it's been given, and then just read them back accurately.
Get the retrieval step wrong and no amount of clever prompting saves you:
the model can only answer from what it's actually shown.

## What's given

- `data/corpus/*.md` -- six short handbook docs (a synthetic "Sandbox
  Handbook" covering port allocation, the hint ladder, the validator
  pass/fail contract, data generation rules, Python tooling conventions,
  and the capstone checkpoint structure). Generated by
  `uv run python generate.py` at the module root; also obtainable in
  memory via `generate.build_rag_corpus(SEED)`, which returns
  `(docs, qa)` -- `docs` is the list you need, each entry
  `{doc_id, title, path, text}`.
- `harness/llm.py` -- the shared `LLMClient`: `client.embed(texts) ->
  list[list[float]]` (batched), `client.generate(prompt, ...) -> str`, and
  a top-level `cosine(a, b) -> float` helper.
- `src/rag.py` -- the scaffold you implement. Three functions, each with a
  docstring spelling out the exact contract: `build_index`, `retrieve`,
  `answer`.

## What's required

Implement all three functions in `src/rag.py`:

1. `build_index(docs, client)` -- chunk the docs (your own chunking
   strategy) and embed the chunks, once. Every chunk must retain its
   source `doc_id`.
2. `retrieve(index, question, client, k)` -- embed the question, rank
   chunks by cosine similarity, return the top-k. Each returned chunk must
   expose its `doc_id`.
3. `answer(question, retrieved, client)` -- generate an answer grounded in
   the retrieved chunks, returning `{"answer": str, "citations":
   list[doc_id]}`.

The validator calls these three functions itself, once per QA pair, using
its own held gold -- there's nothing in `data/` you need to grade yourself
against.

## Completion criteria

From the module root:

```bash
uv run python 05-mini-rag/tests/validate.py
```

The validator runs your pipeline against 15 QA pairs and checks two
metrics:

- **`hit@k`** (primary) -- fraction of questions whose gold source doc
  appears among the top-k chunks `retrieve` returned. This is the
  deterministic signal: did you find the right doc at all.
- **Answer-fact rate** (secondary) -- fraction of questions where the
  generated answer actually contains the fact being asked about. Noisier,
  since it depends on the model's phrasing, so its bar is lower.

Prints `PASSED` with both measured rates, or `NOT PASSED: <reason>` and
exits 1 -- including while `src/rag.py` is still unimplemented
(`NotImplementedError` surfaces as a clean message, no traceback). Needs
Ollama running (`docker compose up -d` at the module root, models pulled)
-- the validator checks this first and tells you exactly what's missing if
not.

## Topics to read up on

- Retrieval-augmented generation (RAG) -- why grounding a model's answer in
  retrieved text beats relying on parametric memory alone
- Text chunking strategies -- fixed-size windows vs. semantic/paragraph
  splits, and the precision/recall tradeoff of chunk size
- Embedding-based semantic search -- cosine similarity over dense vectors,
  and why it finds a paraphrase a keyword search would miss
- Prompt grounding -- instructing a model to answer only from provided
  context, and why that matters for citation accuracy
- top-k retrieval and why k is a tunable tradeoff (too small misses
  relevant context, too large drowns the model in irrelevant text)
