Every Parquet row group carries, in the file footer, the min and max value of every column within that row group — for free, no index you have to build separately. A reader that knows what it's looking for can compare the predicate against those min/max bounds and decide, without decompressing a single byte of actual data, whether that row group could possibly contain a match.

That only helps if the min/max range is narrow. Think about what determines how narrow a row group's `captured_at` range is: if rows land in a row group in whatever order they happened to arrive from the scraper (interleaved across all 18 months, all 40 sources), every row group's min/max will span almost the entire dataset — the statistics are technically present but useless. If rows are sorted by the columns you're going to filter on before they're grouped into row groups, each row group's range collapses to a narrow slice, and most row groups become provably irrelevant to any given query.

Row group *size* interacts with this too, in the opposite direction from what you might expect at first — think about what happens to the min/max range of a sorted column as you make row groups bigger vs smaller, and what that implies for how many groups a probe query can skip.
