`pyarrow.parquet.write_table` and `ParquetWriter.write_table` both accept a `row_group_size` argument — but note it caps the size of row groups produced *from that one call*; it does not accumulate across separate calls to reach the target. If you call `write_table` repeatedly with small batches, you'll get one (or more) row groups per call sized by whatever that call's batch happened to be, not by your intended `row_group_size`. To actually hit a target row group size across a whole streamed file, you need to buffer rows yourself until you have exactly (or up to) that many, then hand the writer one table of that size.

To read back what you wrote and see if it worked: `ParquetFile(path).metadata.row_group(i).column(j).statistics` gives you `.min`, `.max`, and `.has_min_max` for row group `i`, column `j` (find `j` by name via `ParquetFile(path).schema_arrow.names.index(...)`, don't hardcode the position). Compare those bounds against your probe's predicate range yourself — that's the same logic a real pushdown-aware reader runs internally, and it's exactly what this task's benchmark measures separately from the actual pushdown query.

For the "why does a global sort beat a per-chunk sort" question: sorting each 200k-row chunk independently before writing gives you 200k-row-wide `captured_at` ranges repeated across every row group, all overlapping — no better than unsorted, in the worst case. Only a sort across the *entire* dataset produces row groups whose ranges are disjoint slices of the full range.
